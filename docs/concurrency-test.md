# 并发转账正确性测试

本文档描述钱包服务的并发安全性测试，验证在高并发场景下转账操作的正确性。

## 测试目标

验证以下并发安全性要求：

1. **余额守恒**：转账前后所有钱包总余额不变
2. **无负余额**：余额不足时转账失败，不会出现负余额
3. **无竞态条件**：并发操作不会导致数据竞争和数据不一致

2. **并发执行**：多个 goroutine 同时执行转账操作
3. **最终状态验证**：检查所有钱包余额，验证守恒性

## 实现原理

### 并发安全保证

服务层使用 `sync.Mutex` 保护共享数据：

```go
type WalletService struct {
    repo  WalletRepository
    mutex sync.Mutex  // 保护并发访问
}

func (s *WalletService) Transfer(req *TransferRequest) (*TransferResponse, error) {
    s.mutex.Lock()
    defer s.mutex.Unlock()
    
    // 1. 检查余额
    // 2. 扣除金额
    // 3. 增加金额
    // 4. 返回结果
}
```

### 转账事务性

转账操作保证原子性：

```
┌─────────────────────────────────────────┐
│           转账操作流程                    │
├─────────────────────────────────────────┤
│                                         │
│  1. 获取锁                              │
│  2. 检查转出钱包余额                     │
│     └─ 余额不足 → 返回错误，释放锁       │
│  3. 扣除转出钱包金额                     │
│  4. 增加转入钱包金额                     │
│  5. 释放锁                              │
│  6. 返回成功                            │
│                                         │
│  任何步骤失败都会回滚，保证数据一致性     │
└─────────────────────────────────────────┘
```

## 常见问题

### Q: 测试钱包的初始余额是多少？

A: 每个测试钱包初始余额为 1000，测试会创建 `并发数 × 2` 个钱包。

### Q: 如何判断测试是否通过？

A: 检查报告中的三个关键指标：
- 余额守恒: true
- 无负余额: true  
- 竞态条件: 未检测到

### Q: 为什么有些转账失败？

A: 在高并发场景下，如果某个钱包余额被多个线程同时消费，可能出现余额不足的情况。这是预期行为，验证了"无负余额"的正确性。
